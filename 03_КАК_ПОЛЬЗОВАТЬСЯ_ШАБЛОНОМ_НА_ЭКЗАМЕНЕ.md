## 1. Что делать сразу после клона

### Шаг 1.1 — Склонировать репозиторий

```powershell
cd C:\
git clone ССЫЛКА_НА_РЕПО demoexam
cd C:\demoexam
```

Если папка уже создана:

```powershell
cd C:\demoexam
```

### Шаг 1.2 — Установить зависимости

Если используешь обычный `venv`:

```powershell
py -m venv .venv
Set-ExecutionPolicy -Scope Process -ExecutionPolicy Bypass
.\.venv\Scripts\activate
pip install -r requirements.txt
```

Если используешь `uv`:

```powershell
uv sync
```

### Шаг 1.3 — Создать базу PostgreSQL

Открыть `psql`:

```powershell
& "C:\Program Files\PostgreSQL\16\bin\psql.exe" -U postgres
```

Создать базу:

```sql
CREATE DATABASE demoexam_26;
\q
```

Если на экзамене нужно другое имя базы, создай другое имя и поменяй его в `config/settings.py`.

### Шаг 1.4 — Проверить настройки БД

Файл:

```text
config/settings.py
```

Проверь:

```python
DATABASES = {
    "default": {
        "ENGINE": "django.db.backends.postgresql",
        "NAME": "demoexam_26",
        "USER": "postgres",
        "PASSWORD": "123456",
        "HOST": "localhost",
        "PORT": "5432",
    }
}
```

Если пароль PostgreSQL на экзамене другой, меняешь только `"PASSWORD"`.

## 2. Что в шаблоне можно оставить

Обычно можно оставить:

- структуру Django-проекта;
- `manage.py`;
- папку `config`;
- подключение PostgreSQL;
- `ALLOWED_HOSTS = ["*"]`;
- настройку `MEDIA_URL` и `MEDIA_ROOT`;
- настройку `STATIC_URL`;
- CSRF middleware;
- базовую авторизацию Django;
- идею ролей;
- идею CRUD;
- команду импорта как основу;
- команду очистки фотографий как основу;
- подход с `ImageField` и Pillow;
- папки `static`, `media`, `import_data`;
- общий принцип шаблонов.

Главное: если предметная область другая, меняешь сущности и поля, а не весь проект.

## 3. Что чаще всего придется менять

На экзамене могут поменяться:

- предметная область;
- названия моделей;
- поля моделей;
- связи между таблицами;
- CSV-файлы;
- названия колонок в CSV;
- фильтры;
- сортировки;
- поиск;
- роли;
- цвета;
- шрифт;
- расположение элементов;
- расположение фото товара;
- список полей на формах;
- требования к списку заказов или другой сущности.

Чаще всего ты меняешь эти файлы:

| Файл | Что менять |
|---|---|
| `core/models.py` | таблицы, поля, связи |
| `core/forms.py` | поля форм и валидация |
| `core/views.py` | логика страниц, поиск, фильтры, сортировки, права |
| `core/templates/core/*.html` | внешний вид страниц |
| `static/css/style.css` | цвета, шрифт, расположение |
| `core/management/commands/import_data.py` | импорт новых CSV |
| `config/urls.py` | маршруты, если появились новые страницы |
| `database_schema.sql` | красивый SQL-файл для сдачи |

## 5. Как менять модели

Файл:

```text
core/models.py
```

Модели отвечают за таблицы БД.

### Если поменялась предметная область

Например, было:

```text
Product
Order
Supplier
```

А стало:

```text
Service
Request
Client
Employee
```

Тогда меняешь классы в `models.py`.

### Что обязательно проверить в моделях

- Есть ли первичные ключи.
- Есть ли внешние ключи.
- Где нужна связь `ForeignKey`.
- Где нельзя удалять запись, если она используется.
- Какие поля обязательные.
- Какие поля могут быть пустыми.
- Где нужна уникальность.
- Где нужны ограничения: цена не меньше 0, скидка от 0 до 100 и т.д.

### Примеры полей

Строка:

```python
name = models.CharField("Название", max_length=255)
```

Текст:

```python
description = models.TextField("Описание", blank=True)
```

Цена:

```python
price = models.DecimalField("Цена", max_digits=10, decimal_places=2)
```

Количество:

```python
quantity = models.PositiveIntegerField("Количество", default=0)
```

Связь:

```python
category = models.ForeignKey(Category, on_delete=models.PROTECT)
```

Фото:

```python
photo = models.ImageField(upload_to="products/", null=True, blank=True)
```

### После изменения моделей

```powershell
python manage.py makemigrations
python manage.py migrate
```

Если используешь UV:

```powershell
uv run python manage.py makemigrations
uv run python manage.py migrate
```

## 6. Как менять формы

Файл:

```text
core/forms.py
```

Формы отвечают за то, какие поля пользователь видит при добавлении и редактировании.

### Если поменялись поля модели

Например, в модели добавил поле:

```python
color = models.CharField("Цвет", max_length=100)
```

Тогда в форме нужно добавить:

```python
fields = [
    ...
    "color",
]
```

### Где писать проверки

Цена:

```python
def clean_price(self):
    price = self.cleaned_data["price"]
    if price < 0:
        raise forms.ValidationError("Цена не может быть отрицательной.")
    return price
```

Количество:

```python
def clean_quantity(self):
    quantity = self.cleaned_data["quantity"]
    if quantity < 0:
        raise forms.ValidationError("Количество не может быть отрицательным.")
    return quantity
```

Фото:

```python
def clean_photo(self):
    photo = self.cleaned_data.get("photo")
    if not photo:
        return photo
    try:
        image = Image.open(photo)
        image.verify()
    except Exception as exc:
        raise forms.ValidationError("Загрузите корректное изображение.") from exc
    photo.seek(0)
    return photo
```

## 7. Как менять views

Файл:

```text
core/views.py
```

Views отвечают за логику страниц:

- какие данные взять из БД;
- какие фильтры применить;
- куда перейти после сохранения;
- кому разрешен доступ;
- какой шаблон открыть.

### Список объектов

Список товаров сейчас работает через `ProductListView`.

Если главная сущность поменялась, например стала `Service`, то меняешь:

```python
model = Product
```

на:

```python
model = Service
```

и меняешь шаблон/контекст.

### Поиск

Поиск находится в `get_queryset()`.

Пример:

```python
search = self.request.GET.get("search", "").strip()
if search:
    queryset = queryset.filter(
        Q(name__icontains=search)
        | Q(description__icontains=search)
        | Q(article__icontains=search)
    )
```

Если в задании поиск по другим полям, меняешь поля внутри `Q(...)`.

### Фильтр

Пример фильтра по поставщику:

```python
supplier = self.request.GET.get("supplier")
if supplier:
    queryset = queryset.filter(supplier_id=supplier)
```

Если нужен фильтр по категории:

```python
category = self.request.GET.get("category")
if category:
    queryset = queryset.filter(category_id=category)
```

### Сортировка

Пример:

```python
sort_map = {
    "quantity_asc": "quantity",
    "quantity_desc": "-quantity",
}
```

Если нужна сортировка по цене:

```python
sort_map = {
    "price_asc": "price",
    "price_desc": "-price",
}
```

Папка:


Главные шаблоны:

|---|---|

### Если поменялись поля

Например, в товаре теперь нужно показывать цвет.

В шаблоне списка добавляешь:

```django
<p>Цвет: {{ product.color }}</p>
```

В форме, если используешь Django form:

```django
{{ form.color }}
```

### Если поменялась главная сущность

Например, вместо товаров теперь услуги.

- переименовывать файлы не обязательно;

- переменные можно оставить технически, но лучше привести к новой предметной области, если есть время.

Во всех формах с POST оставляй:

```django
{% csrf_token %}
```

```css
.product-photo {
```

```css
.product-photo {
```

## 10. Как менять импорт CSV

Файл:

```text
core/management/commands/import_data.py
```

Если на экзамене данные дали в Excel `.xlsx`, сначала открой файл в Excel и сохрани как `CSV UTF-8`. Сам скрипт импорта читает только CSV.

Скрипт специально сделан простым, в экзаменационном стиле:

- отдельный метод для пунктов выдачи;
- отдельный метод для товаров;
- отдельный метод для пользователей;
- отдельный метод для заказов;
- внутри каждого метода обычный `open(...)`;
- дальше `csv.DictReader(...)` или `csv.reader(...)`;
- потом цикл `for row in reader`;
- затем `get_or_create(...)`, `update_or_create(...)` или `create(...)`.

То есть это не сложный универсальный импорт, а понятный скрипт, который легко объяснить и быстро переписать.

### Что может поменяться

- названия файлов;
- названия колонок;
- количество файлов;
- порядок колонок;
- предметная область;
- связи между таблицами.

### Где менять названия файлов

Внутри методов ищи строки с `first_file`.

Пример для товаров:

```python
path = self.first_file(base_path, "products.csv", "tovar.csv", "Tovar.csv")
```

Если на экзамене файл называется `services_import.csv`, меняешь или дописываешь:

```python
path = self.first_file(base_path, "services_import.csv")
```

Можно оставить несколько вариантов:

```python
path = self.first_file(base_path, "services.csv", "services_import.csv")
```

### Где менять названия колонок

Пример:

```python
article = self.value(row, "article", "артикул")
```

Если в CSV колонка называется `Код товара`, добавь:

```python
article = self.value(row, "article", "артикул", "код товара")
```

Метод `value(...)` нужен, чтобы скрипт понимал разные названия одной и той же колонки.

Например, все эти варианты могут означать одно поле:

```python
self.value(row, "name", "название", "наименование товара")
```

### Где менять создание объекта

В новом скрипте смотри методы:

```python
import_pickup_points()
import_products()
import_users()
import_orders()
```

Если меняется предметная область, меняешь нужный метод или создаешь новый.

Пример создания справочника:

```python
supplier, _ = Supplier.objects.get_or_create(
    name=self.value(row, "supplier", "поставщик") or "Не указан"
)
```

Пример создания или обновления товара:

```python
Product.objects.update_or_create(
    article=article,
    defaults={
        "name": ...,
        "price": ...,
    },
)
```

Если модель поменялась, меняешь `Product` и поля в `defaults`.

Например, если вместо товаров будут услуги:

```python
Service.objects.update_or_create(
    code=code,
    defaults={
        "name": self.value(row, "name", "название услуги"),
        "price": self.decimal(self.value(row, "price", "цена")),
    },
)
```

### Как устроены 4 главных блока импорта

Пункты выдачи:

```python
with open(path, encoding="utf-8-sig", newline="") as file:
    reader = csv.reader(file, delimiter=";")
    for row in reader:
        address = row[0].strip()
        PickupPoint.objects.get_or_create(address=address)
```

Товары:

```python
with open(path, encoding="utf-8-sig", newline="") as file:
    reader = csv.DictReader(file, delimiter=self.get_delimiter(file))
    for row in reader:
        supplier, _ = Supplier.objects.get_or_create(...)
        Product.objects.update_or_create(...)
```

Пользователи:

```python
with open(path, encoding="utf-8-sig", newline="") as file:
    reader = csv.DictReader(file, delimiter=self.get_delimiter(file))
    for row in reader:
        user, created = User.objects.get_or_create(...)
        if created:
            user.set_password(password)
        user.save()
```

Заказы:

```python
with open(path, encoding="utf-8-sig", newline="") as file:
    reader = csv.DictReader(file, delimiter=self.get_delimiter(file))
    for row in reader:
        order, created = Order.objects.update_or_create(...)
        OrderItem.objects.filter(order=order).delete()
        OrderItem.objects.create(...)
```

### Что менять, если CSV полностью другой

1. Поменять имя файла в `first_file(...)`.
2. Поменять названия колонок в `self.value(...)`.
3. Поменять модель, которую создаешь.
4. Поменять поля в `defaults`.
5. Поменять связи через `ForeignKey`.
6. Запустить импорт и посмотреть ошибки.

Главное: не пытайся сделать идеально универсально. На экзамене лучше простой рабочий импорт под конкретные файлы.

### Проверка импорта

```powershell
python manage.py import_data --path import_data
```

После этого смотри данные:

- через свои страницы;
- через DBeaver;

## 11. Как менять фильтры, поиск и сортировку

Фильтры, поиск и сортировка меняются в двух местах:

```text
core/views.py
core/templates/core/product_list.html
```

### В `views.py`

Там пишется логика:

- какие GET-параметры читать;
- по каким полям искать;
- какую сортировку применять.

### В HTML-шаблоне

Там пишутся поля управления:

- input для поиска;
- select для фильтра;
- select для сортировки.

### Пример: добавить фильтр по категории

В `views.py`:

```python
category = self.request.GET.get("category")
if category:
    queryset = queryset.filter(category_id=category)
```

В контекст:

```python
context["categories"] = Category.objects.all()
```

В шаблон:

```django
<select name="category" onchange="this.form.submit()">
    <option value="">Все категории</option>
    {% for category in categories %}
        <option value="{{ category.id }}">{{ category.name }}</option>
    {% endfor %}
</select>
```

## 12. Как менять роли и доступ

Роли находятся в:

```text
core/models.py
core/views.py
```

Сейчас роли:

```text
admin
manager
client
```

Гость — это пользователь без входа.

### Если появилась новая роль

В `Role` добавь:

```python
OPERATOR = "operator"
```

И в `ROLE_CHOICES`:

```python
(OPERATOR, "Оператор")
```

Потом сделать миграции:

```powershell
python manage.py makemigrations
python manage.py migrate
```

И поправить доступ во views.

### Если поменялись права

Ищи в `views.py`:

```python
allowed_roles = (...)
```

или проверки:

```python
user_role(self.request.user) == Role.ADMIN
```

Меняешь, кому что разрешено.

## 13. Что делать с фотографиями

Если в новом задании есть изображения:

- оставляй `ImageField`;
- оставляй папку `media`;
- оставляй Pillow;
- оставляй удаление старых фото;
- оставляй `cleanup_media`.

Если фото должны быть у другой модели, например не у товара, а у сотрудника, переносишь поле:

```python
photo = models.ImageField(upload_to="employees/", null=True, blank=True)
```

И меняешь команду очистки, чтобы она смотрела другую папку.

Если изображения вообще не требуются, можно не использовать поля фото на новых страницах, но вариативный пункт про Pillow тогда может не закрыться.

## 14. Что делать с БД, ERD и SQL

### После изменения моделей

```powershell
python manage.py makemigrations
python manage.py migrate
```

### ERD в DBeaver

В ERD включай только таблицы предметной области.

Для этого шаблона это:

```text
core_role
core_user
core_supplier
core_category
core_manufacturer
core_pickuppoint
core_product
core_order
core_orderitem
```

Если предметная область поменялась, включай новые `core_*` таблицы.

Служебные таблицы Django обычно не включай:

```text
auth_*
django_*
core_user_groups
core_user_user_permissions
```

### SQL-файл через pg_dump

Структура без данных:

```powershell
& "C:\Program Files\PostgreSQL\16\bin\pg_dump.exe" -U postgres -s demoexam_26 > database_schema_dump.sql
```

Структура с данными:

```powershell
& "C:\Program Files\PostgreSQL\16\bin\pg_dump.exe" -U postgres demoexam_26 > database_full_dump.sql
```

### Красивый SQL-файл

Файл:

```text
database_schema.sql
```

Его можно вручную привести к аккуратному виду:

- комментарии перед таблицами;
- только таблицы предметной области;
- понятные `CREATE TABLE`;
- стартовые роли;
- тестовые пользователи;
- тестовые данные.
