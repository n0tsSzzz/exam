## 4. Предметная область ПУ

В тренировочном варианте предметная область связана с продажей обуви.

Система должна хранить и показывать:

- пользователей;
- роли пользователей;
- товары;
- категории товаров;
- производителей;
- поставщиков;
- пункты выдачи;
- заказы;
- состав заказов;
- фотографии товаров.

Основной пользовательский сценарий:

1. Пользователь открывает приложение.
2. Сначала видит окно входа.
3. Может войти по логину и паролю или как гость.
4. После входа видит товары.
5. Возможности зависят от роли.
6. Менеджер работает со списками и заказами.
7. Администратор управляет товарами и заказами.

## 5. Модуль 1 — база данных

### 5.1. Что нужно сделать

- Проанализировать предметную область.
- Выделить сущности.
- Выделить связи.
- Создать базу данных в СУБД.
- Для нашей подготовки использовать PostgreSQL.
- Привести БД к 3 нормальной форме.
- Обеспечить ссылочную целостность.
- Задать первичные ключи.
- Задать внешние ключи.
- Загрузить данные из файлов с пометкой `import`.
- Получить ER-диаграмму.
- Сохранить SQL-скрипт базы или другой файл конфигурации с данными.

### 5.2. Таблицы предметной области

Для нашего Django-шаблона предметные таблицы:

| Таблица | Назначение |
|---|---|
| `core_role` | роли пользователей |
| `core_user` | пользователи |
| `core_supplier` | поставщики |
| `core_category` | категории |
| `core_manufacturer` | производители |
| `core_pickuppoint` | пункты выдачи |
| `core_product` | товары |
| `core_order` | заказы |
| `core_orderitem` | состав заказов |

### 5.3. Служебные таблицы Django

После миграций в DBeaver появятся дополнительные таблицы:

```text
auth_group
auth_group_permissions
auth_permission
core_user_groups
core_user_user_permissions
django_content_type
django_migrations
django_session
```

Это нормально. Django создает их для:

- групп;
- прав доступа;
- сессий;
- истории миграций;
- связи пользователя с группами и правами.

### 5.4. Что включать в ER-диаграмму

В ER-диаграмму предметной области включай таблицы проекта:

```text
core_role
core_user
core_supplier
core_category
core_manufacturer
core_pickuppoint
core_product
core_order
core_orderitem
```

Служебные таблицы `auth_*`, `django_*`, `core_user_groups`, `core_user_user_permissions` обычно не включают, если эксперт просит схему именно предметной области.

### 5.5. Требования к ER-диаграмме

- Формат сдачи: PDF.
- Должны быть таблицы.
- Должны быть связи.
- Должны быть атрибуты.
- Должны быть первичные ключи.
- Должны быть внешние ключи.
- Типами данных можно пренебречь, если в задании это разрешено.

## 7. Авторизация и роли

### 7.1. Стартовое окно

При запуске первым открывается окно входа.

На нем должны быть:

- поле логина;
- поле пароля;
- кнопка входа;
- кнопка входа как гость;
- сообщения об ошибках.

### 7.2. После входа

После успешной авторизации:

- пользователь попадает в приложение;
- получает доступ к функциям по своей роли;
- справа сверху отображается ФИО пользователя;
- есть возможность выйти обратно на экран входа.

### 7.3. Роли

| Роль | Возможности |
|---|---|
| Гость | просмотр товаров без поиска, фильтрации и сортировки |
| Клиент | просмотр товаров без поиска, фильтрации и сортировки |
| Менеджер | просмотр товаров с поиском, фильтром, сортировкой; просмотр заказов |
| Администратор | полный CRUD товаров и заказов |

### 7.4. Что нельзя показывать лишним ролям

Гостю и клиенту нельзя показывать:

- поиск;
- фильтр;
- сортировку;
- кнопку добавления товара;
- кнопки редактирования и удаления товара;
- раздел заказов.

Менеджеру нельзя показывать:

- добавление товара;
- редактирование товара;
- удаление товара;
- добавление заказа;
- редактирование заказа;
- удаление заказа.

## 8. Список товаров

- заглушка, если фото нет;

- поставщик;

## 9. Модуль 3 — интерфейс и товары

### 9.1. Общий интерфейс

Требования:

- интерфейс должен быть последовательным;
- должны быть переходы между страницами;
- должна быть кнопка назад, где она нужна;
- каждая страница имеет заголовок;
- ошибки не должны ломать приложение;
- пользователь получает понятные сообщения;
- необратимые операции требуют предупреждения.

### 9.2. Поиск

Доступен менеджеру и администратору.

Требования:

- поиск по всем доступным текстовым данным товара;
- поиск работает по нескольким атрибутам;
- поиск происходит в реальном времени;
- отдельная кнопка поиска не обязательна;
- поиск работает вместе с фильтром.

Примеры текстовых полей:

- название;
- описание;
- категория;
- производитель;
- поставщик;
- артикул, если он есть.

### 9.3. Фильтрация

Доступна менеджеру и администратору.

Требования:

- фильтр по поставщику;
- первый элемент: `Все поставщики`;
- при выборе `Все поставщики` фильтр сбрасывается;
- фильтр применяется вместе с поиском;
- сортировка сохраняется после фильтрации.

### 9.4. Сортировка

Доступна менеджеру и администратору.

Требования:

- сортировка по количеству на складе;
- по возрастанию;
- по убыванию;
- выбранная сортировка сохраняется при поиске и фильтрации.

### 9.5. Добавление товара

Доступно только администратору.

На форме должны быть:

- фото;
- наименование;
- категория;
- описание;
- производитель;
- поставщик;
- цена;
- единица измерения;
- количество;
- скидка.

Ограничения:

- категория выбирается из выпадающего списка;
- производитель выбирается из выпадающего списка;
- поставщик выбирается из выпадающего списка;
- цена может иметь сотые части;
- цена не может быть отрицательной;
- количество не может быть отрицательным;
- скидка от `0` до `100`.

### 9.6. Редактирование товара

Доступно только администратору.

Требования:

- переход к редактированию из списка товаров;
- форма открывается отдельно;
- все поля заполняются текущими данными;
- ID при добавлении не отображается;
- ID при редактировании только для чтения, если он показан;
- после сохранения список обновляется.

### 9.7. Фото товара

Требования:

- администратор может добавить фото;
- администратор может заменить фото;
- фото сохраняется в папку приложения;
- в БД хранится путь к фото;
- размер фото ограничивается до `300x200` пикселей;
- при замене старое фото удаляется;
- если фото нет, показывается заглушка.

### 9.8. Удаление товара

Доступно только администратору.

Требования:

- перед удалением есть подтверждение;
- товар, который есть в заказе, удалить нельзя;
- пользователь получает понятное сообщение;
- после удаления список обновляется;
- фото удаленного товара удаляется из папки.

## 11. Руководство по стилю

### 11.1. Общие требования

- Все экранные формы имеют заголовок.
- На главной форме есть логотип.
- Логотип берется из ресурсов.
- Логотип нельзя искажать.
- Нельзя менять пропорции логотипа.
- Нельзя менять цвет логотипа.
- Для приложения используется иконка, если это возможно.
- Шрифт: `Times New Roman`.

### 11.2. Цвета

| Назначение | Цвет |
|---|---|
| Основной фон | `#FFFFFF` |
| Дополнительный фон | `#7FFF00` |
| Акцент / целевое действие | `#00FA9A` |
| Скидка больше 15% | `#2E8B57` |

### 11.3. Практическое применение

- Основной фон страницы: белый.
- Шапка или второстепенные блоки: `#7FFF00`.
- Основные кнопки: `#00FA9A`.
- Строки товаров со скидкой больше 15%: `#2E8B57`.
- Товары без остатка: голубой фон.
- Старая цена со скидкой: красная и перечеркнутая.
- Итоговая цена: черная.

## 12. Вариативная часть

Вариативная часть нашего колледжа: 25 баллов.

### 12.1. Скрипт загрузки CSV в БД — 5 баллов

В проекте должна быть management-команда:

```text
core/management/commands/import_data.py
```

Команда должна:

- загружать данные из CSV;
- если данные выдали в Excel, сначала сохранить их как CSV UTF-8;
- создавать данные в БД;
- не создавать дубли;
- не падать на пустых строках;
- выводить понятный результат импорта.

Запуск:

```powershell
python manage.py import_data --path import_data
```

### 12.2. CSRF-защита — 5 баллов

Должно быть:

- `CsrfViewMiddleware` в `settings.py`;
- `{% csrf_token %}` во всех POST-формах;
- вход, выход, создание, редактирование и удаление работают через POST там, где это действие изменяет данные.

Проверить формы:

- login;
- logout;
- product create;
- product update;
- product delete;
- order create;
- order update;
- order delete.

### 12.3. Удаление неиспользуемых фотографий — 5 баллов

Должно быть:

- удаление старого фото при замене товара;
- удаление фото при удалении товара;
- отдельная команда очистки лишних файлов.

Команда:

```text
core/management/commands/cleanup_media.py
```

Запуск:

```powershell
python manage.py cleanup_media
```

### 12.4. Pillow — 5 баллов

Pillow используется для:

- проверки изображения;
- обработки загруженного фото;
- изменения размера до `300x200`;
- сохранения оптимального файла.

Зависимость:

```text
pillow
```

### 12.5. Django — 5 баллов

Проект должен быть Django-приложением:

- есть `manage.py`;
- есть `config/settings.py`;
- есть приложение `core`;
- есть модели;
- есть миграции;
- есть views;
- есть templates;
- есть static/media;
- проект запускается через `python manage.py runserver`.

## 16. Техника безопасности

### Перед началом

- пройти инструктаж;
- убедиться, что нет противопоказаний по здоровью;
- проверить положение монитора;
- проверить положение клавиатуры;
- убедиться, что кабели находятся с тыльной стороны рабочего места;
- убедиться, что оборудование загрузилось корректно.

### Во время работы

- не закрывать вентиляционные отверстия;
- соблюдать инструкции эксплуатации оборудования;
- соблюдать регламентированные перерывы;
- не отключать и не подключать интерфейсные кабели;
- не прикасаться к задней панели системного блока при включенном питании;
- не допускать попадания влаги и грязи на технику;
- не вскрывать и не ремонтировать оборудование самостоятельно;
- при неисправности прекратить работу и сообщить эксперту.

### После работы

- завершить задачи на ПК;
- корректно выключить оборудование;
- привести рабочее место в порядок.
